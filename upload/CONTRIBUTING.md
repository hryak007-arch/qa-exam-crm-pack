# Contributing

Thanks for your interest in improving this project.

## Basic rules
1. Create a separate branch for each change.
2. Keep commits small and meaningful.
3. Do not commit secrets, local databases, or generated coverage reports.
4. Update the README when behavior or setup changes.
5. Run tests and linting before opening a pull request.

## Recommended workflow
1. Fork or clone the repository.
2. Create a branch:
   `git checkout -b feature/short-description`
3. Make your changes.
4. Commit clearly:
   `git commit -m "Add: short description of change"`
5. Push the branch and open a pull request.

## Commit message style
Use one of these prefixes:
- `Add:` for new features
- `Fix:` for bug fixes
- `Docs:` for documentation
- `Refactor:` for code cleanup
- `Test:` for tests
- `Chore:` for repository maintenance

## Pull request checklist
- [ ] The project still starts correctly
- [ ] No private data was added
- [ ] README or docs were updated if needed
- [ ] Unused files were removed
- [ ] Tests were run locally
